🎭 What is Playwright?

Playwright is a modern, open-source browser automation library developed by Microsoft that enables reliable end-to-end testing across all major browsers.

Key Features

  • Cross-browser support - Works seamlessly on Chromium, Firefox, and WebKit
  • Multiple languages - Available in JavaScript, Python, .NET, and Java
  • Fast and reliable - Auto-waits for elements, built-in retry logic
  • Network control - Intercept and modify network requests
  • Full context isolation - Lightweight isolation for parallel testing
  • Inspector & Debugging - Built-in tools for test development and debugging

Why Playwright for API Testing?

💡 Perfect Foundation: While Playwright excels at browser automation, the Xtrem Framework leverages Playwright's powerful infrastructure - particularly its context and fixture capabilities - to create a robust, type-safe API testing solution with GraphQL support.

🏗️ Framework Architecture

Project Structure Overview

services/api-test ├── docs/ # Documentation and onboarding guides ├── lib/ # Shared typed test API, helpers, utilities │ ├── index.ts # Composite GraphApi type + exported Playwright test fixture │ ├── functions/ # Generic cross-domain helper functions │ ├── reporters/ # Custom Playwright reporters │ ├── test-utils/ # Test-specific reusable helpers │ ├── types/ # Shared TypeScript types │ ├── [domain]/ # Domain helper classes (CRUD/query flows) │ │ └── index.ts # Domain exports ├── tests/ # Executable spec files grouped by domain │ ├── global.setup.ts # Pre-test checks/bootstrap (setup project) │ ├── global.teardown.ts # Post-test cleanup (cleanup project) │ ├── [domain]/ │ │ └── *.spec.ts # Domain test suites └── playwright.config.ts # Project dependency orchestration and runtime config

Four Key Layers

1. Fixtures Layer - Provides typed GraphQL access with full type safety
2. Helper Classes - Domain-specific operations extending BaseNode
3. Utility Functions - Pure functions for test data generation
4. Test Layer - Actual test implementations using all above

▶️ Running Tests Effectively

Essential Test Commands

# Build the project pnpm build # Type check pnpm typecheck # Run all tests npx playwright test tests/ # Debug mode with server logging PLAYWRIGHT_SERVER_LOG=true pnpm test # Quick mode (skip layer loading) PLAYWRIGHT_SKIP_LAYER_LOAD=true pnpm test # Run only release tests pnpm test --grep=@release # Run only development tests pnpm test --grep=@development # Run only flaky tests pnpm test --grep=@flaky # Run all tests on chromium pnpm test --project=chromium # Run API tests with custom timeout PLAYWRIGHT_SERVER_TIMEOUT=300000 xtrem test --api # Run remote tests ./scripts/test-remote.sh # Run tests without server START_SERVER=false xtrem test --api # Run specific file npx playwright test tests/master-data/customer.spec.ts # Watch mode for development npx playwright test --watch

Important Environment Variables

  • PLAYWRIGHT_SKIP_LAYER_LOAD=true - Skip test data layer loading (faster)
  • PLAYWRIGHT_GRAPH_LOG=true - Enable GraphQL request/response logging
  • DEBUG=pw:webserver - Enable server debug output
  • CI=true - Activates CI mode (affects retries and workers)

✍️ Writing Your First Test

Basic Test Structure

This example shows a complete CRUD flow for a Customer entity. The goal is to demonstrate the standard structure you should follow for most API tests in this framework.

  • Imports: Bring in Playwright assertions, the typed test fixture, and your domain helper class.
  • Test tag: @development controls where the test runs in pipelines.
  • Helper usage: MasterData.Customer centralizes GraphQL operations so tests stay readable.
  • Steps: Split the flow into create/update/delete for clean reporting and easier debugging.
  • Assertions: Validate critical outputs after each action, not only at the end.
import { expect } from '@playwright/test'; import { test } from '../../lib'; import { MasterData } from '../../lib/master-data'; test.describe('Customer', () => { test('@development Customer CRUD', async ({ graph }) => { const customerClass = new MasterData.Customer(graph); let customer; await test.step('Create customer', async () => { customer = await customerClass.create(); expect(customer._id).toBeDefined(); }); await test.step('Update customer', async () => { const updated = await customerClass.update({ name: 'Updated Name' }); expect(updated.name).toBe('Updated Name'); }); await test.step('Delete customer', async () => { await customerClass.delete({ _id: customer._id }); }); }); });
💡 How to read this quickly: Start from the test title and tag, then follow each test.step() in order. If one step fails, the report tells you exactly which business action broke.
⚠️ Remember: Always use test.step() for better organization and reporting. Tag your tests appropriately (@development, @release, @flaky).

🛠️ Creating Helper Classes

BaseNode Extension Pattern

Helper classes extend BaseNode to provide domain-specific operations with full type safety.

import { BaseNode } from '@sage/xtrem-playwright'; import { randomString } from '../functions/random'; export class PartnerClass extends BaseNode< GraphApi, Partner, PartnerInput, PartnerType > { get nodeName() { return '@sage/xtrem-master-data/Partner'; } get defaultSelector() { return { _id: true, code: true, name: true, isActive: true }; } async getRandomData(createData?: PartnerInput) { const code = randomString(6).toUpperCase(); return { code, name: `Partner ${code}`, isActive: true, ...createData }; } }

Benefits

  • Automatic CRUD operations (create, read, update, delete)
  • Random data generation for tests
  • Type-safe GraphQL operations
  • Reusable across multiple test files

🏷️ Test Tagging Strategy

The Three Standard Tags

@development
For feature development, integration testing, and core functionality verification. These tests run during active development cycles.
@release
Stable, production-ready tests that verify API contracts. These must always pass in release builds and are critical for deployments.
@flaky
Tests with known intermittent failures, external dependencies, or timing issues. These are monitored separately while fixes are developed.

Usage Examples

// Development test test('@development Customer creation', async ({ graph }) => { // Test in development }); // Release test test('@release Core API validation', async ({ graph }) => { // Stable production test }); // Run specific tags pnpm test --grep @release

Pipeline Architecture

The testing framework includes Azure DevOps YAML pipeline templates for seamless CI/CD integration:

Project pipeline orchestrating agent startup, Playwright execution, MySQL export, Teams notifications, and Jira ticket creation.

Stage Flow (Current Pipeline)

  • StartAgent: starts AWS EC2 instance before tests run
  • Run_Playwright_Template: executes Playwright tests with tenant provisioning
  • WriteToMysql: exports results when enabled for manual/scheduled runs
  • NotifyStatus: sends Teams success/failure notifications
  • JiraTaskCreation: creates a Jira ticket when Playwright fails and ticketing is enabled

Key Pipeline Features

  • Automated Environment Setup: Handles tenant provisioning and cleanup
  • Tag-Based Execution: Configure via PLAYWRIGHT_GREP parameter (@development, @release, @flaky)
  • Parallel Testing: Configurable worker instances for faster execution
  • Multiple Cluster Support: Test across different environments and configurations
  • Authentication Handling: Secure and unsecure auth methods
  • Result Reporting: Comprehensive test result parsing and reporting

Pipeline Configuration Parameters

# Key parameters for pipeline execution PLAYWRIGHT_GREP: '@development' # Filter tests by tag XTREM_TEST_MAX_INSTANCES: 4 # Parallel worker count (default) TARGET_URL: 'https://...' # Target environment TENANT_PROVISIONING_METHOD: 'static' # Tenant creation method writeSQL: 'Yes|No' # Enable MySQL export stage sendTeamsNotification: 'Yes|No' # Enable Teams notification stage createJiraTicket: 'Yes|No' # Enable Jira ticket creation
💡 CI Best Practice: Use fully parallel execution with appropriate worker limits. Tests are designed for isolation and can run concurrently for faster feedback.

✅ Best Practices

1. Test Independence

Each test must be completely independent and create its own data. Never rely on data from other tests or shared global state.

2. Use Test Steps

Break tests into logical steps for better reporting and debugging. Each step should represent a meaningful action.

3. Meaningful Assertions

// ✅ GOOD - Clear assertions with messages expect(customer._id, 'Customer ID should exist').toBeDefined(); expect(customer.isActive, 'Should be active by default').toBe(true); // ❌ POOR - Generic assertions expect(customer).toBeDefined();

4. Clean Up Test Data

Always delete created test data, especially in @release tests. Use try-finally blocks to ensure cleanup happens even if tests fail.

5. Leverage Helper Classes

Use domain helper classes instead of raw graph queries for better maintainability and reusability.

🔧 Troubleshooting Guide

Common Issues & Solutions

Issue: Tests fail with "Cannot query field"
Solution: Verify the field exists in GraphQL schema and check your selector matches the API definition.
Issue: "No _id to update" error
Solution: Ensure entity is created before updating. The helper class needs an entity instance.
Issue: Layer load takes too long
Solution: Use PLAYWRIGHT_SKIP_LAYER_LOAD=true pnpm test for faster development iterations.

Debugging Tools

# Enable GraphQL logging PLAYWRIGHT_GRAPH_LOG=true pnpm test # Full debug mode with server logging PLAYWRIGHT_SERVER_LOG=true PLAYWRIGHT_GRAPH_LOG=true pnpm test # Run single test with verbose output npx playwright test tests/master-data/customer.spec.ts --debug
